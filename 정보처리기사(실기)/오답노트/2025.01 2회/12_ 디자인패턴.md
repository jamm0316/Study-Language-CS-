## 12. 다음 설명에서 괄호에 들어갈 알맞은 디자인 패턴을 아래에서 찾아 쓰시오.

| 생성패턴             | 구조패턴      | 행위패턴                   |
| ---------------- | --------- | ---------------------- |
| Abstract Factory | Adapter   | Chain of Reponsibility |
| Builder          | Bridge    | Command                |
| Factory Method   | Composite | Interpreter            |
| Prototype        | Decorator | Iterator               |
| Singleton        | Facade    | Mediator               |
|                  | Proxy     | Memento                |
|                  |           | Observer               |
|                  |           | Strategy               |
|                  |           | Template Method        |

( )은 복잡한 시스템을 개발하기 쉽도록 클래스나 객체들을 조회하는 패턴에 속하며 대화자라는 이름으로도 불린다.
내부에서는 객체간의 복잡한 관계를 단순하게 정리해주고, 외부에서는 객체의 세부적인 내용을 숨기는 역할을 한다.

<br><br><br><br><br>

sol) 헷갈리는 패턴
- 추상 팩토리(Abstract Factory): 구체적인 클래스에 의존하지 않고, 인터페이스를 통해 서로 연관·의존하는 객체들의 그룹으로 생성하여 추상적으로 표현
- 팩토리메소드(Factory Method): 객체 생성을 서브클래스에서 처리하도록 캡슐화한 패턴
- 어댑터(Adapter): 호환성이 없는 클래스들의 인터페이스를 다른 클래스가 이용할 수 있도록 변환해주는 패턴
- 퍼사드(Facade): 복잡한 서브클래스들을 피해 상위 인터페이스를 구성하여 서브클래스를 간편하게 이용
- 브릿지 패턴(Bridge): 구현부에서 추상층을 분리하여 각자 독립적으로 변형이 가능하고 확장이 가능하도록 한다. 즉 기능과 구현에 대해서 두 개의 별도의 클래스로 구현한다.

- **생성 패턴: 팩토리 패턴 차이점 비교** <br>

| 구분    | Factory Method        | Abstract Factory       |
| ----- | --------------------- | ---------------------- |
| 목적    | 하나의 객체 생성을 서브클래스에 위임  | 연관된 객체들의 집합(제품군)을 생성   |
| 결정 주체 | 서브클래스가 어떤 클래스를 만들지 결정 | 팩토리 자체가 어떤 제품군을 만들지 결정 |
| 규모    | 단일 제품                 | 제품 패밀리(여러객체)           |

- **구조 패턴: 어댑터, 파사드, 브릿지, 프록시 차이점 비교** <br>

| 구분    | Adapter     | Facade     | Bridge      | Proxy       |
| ----- | ----------- | ---------- | ----------- | ----------- |
| 목적    | 인터페이스 호환    | 복잡성 은닉     | 추상·구현 분리    | 접근 제어/대리    |
| 관계 방향 | 기존 ↔ 신규 연결  | 내부를 감쌈     | 양측을 독립 확장   | 원본을 대체      |
| 핵심 질문 | “어떻게 연결하지?” | “어떻게 숨기지?” | “어떻게 분리하지?” | “어떻게 대리하지?” |
